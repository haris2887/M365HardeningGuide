Critical Controls to Start
Stryker Incident — March 11, 2026: An Iran-linked threat group called Handala (assessed as Void Manticore / Storm-0842, MOIS-affiliated) compromised Stryker Corporation's M365 tenant and used Microsoft Intune's native remote wipe to execute a mass device wipe across 79 countries. Attackers abused the cloud management plane itself — a legitimate MDM feature — making it invisible to endpoint security tools.
Critical Insight: Traditional endpoint-only controls cannot detect this attack type. Actions came through authorized cloud management APIs. Detection must extend to the identity and cloud admin layers — not just endpoints.

THE CLOUD CONTROL PLANE

Microsoft Intune
= Group Policy + SCCM
Manages device config, software deployment, compliance enforcement, and remote wipe. A compromised Intune Admin can push destructive actions to every managed endpoint.
Microsoft Entra ID
= The Domain Controller
Identity authority for the entire tenant. Controls authentication, roles, Conditional Access, and federation trusts. Owning Entra ID = owning the cloud equivalent of Active Directory.
Microsoft Graph API
= Programmatic Admin Console
Provides programmatic access to virtually every M365 admin function. An attacker with the right permissions can script tenant-wide destructive actions at scale via authenticated REST calls.
OAuth App Registrations
= Service Accounts with Delegated Power
OAuth apps and service principals hold persistent, non-interactive permissions. A compromised app registration with device management permissions = rogue Domain Admin that never needs a password.

4 Misconfigurations

Most Microsoft tenants that suffer destructive attacks share these four weaknesses. All four present = single phished admin credential is enough.

01
Too Many Accounts Have Device Wipe Permissions
Default Intune roles include destructive permissions (Wipe, Retire, Factory Reset, Delete). Any compromised account can script tenant-wide destructive actions via Graph API.
→ FIX: Custom RBAC roles without wipe permissions. Limit wipe capability to 2 break-glass accounts only. (Controls 8, 9)
02
Admin Access Allowed from Unmanaged Devices
Administrators can access the Intune portal from any device, anywhere. Attackers who steal credentials log in from their own infrastructure and execute destructive actions.
→ FIX: Conditional Access requiring compliant devices + phishing-resistant MFA + named network locations for all admin portal access. (Controls 5, 6)
03
Permanent Administrative Role Assignments
Privileged roles like Intune Administrator and Global Administrator are often permanently assigned. A compromised credential immediately grants destructive capabilities with no approval gate.
→ FIX: PIM with eligible-only assignments, MFA, approval workflows, and 1-hour activation windows. (Control 2)
04
OAuth Application Consent Enabled for Users
When user consent is enabled, attackers can trick users into granting API access to malicious applications — including permissions that allow device management operations.
→ FIX: Disable user consent tenant-wide. Require admin approval for all application permissions. (Controls 12, 13, 14)

Identity & Access Controls 1–7
1
Enforce MFA Appropriately by Role
CriticalIdentity
4 hrs

Not all MFA is equal. Admin accounts need phishing-resistant MFA (FIDO2 hardware keys), while regular users can use standard push MFA with number matching. AiTM phishing kits like EvilGinx intercept passwords and push MFA tokens in real-time. FIDO2 keys are bound to the legitimate domain and cannot be phished this way.

Admin Roles (Global Admin, Intune Admin, Cloud Device Admin, Security Admin)
Entra ID → Security → Authentication Methods → FIDO2 Security Key → Enable
Conditional Access → New Policy:
Name: Require phishing-resistant MFA for admins
Users: Directory roles → [all admin roles]
Cloud apps: All cloud apps
Grant: Require authentication strength → Phishing-resistant MFA
Regular Users
Entra ID → Security → Authentication Methods → Microsoft Authenticator
Number matching: Enabled (required)
Show additional context: Enabled
Conditional Access → New Policy:
Name: Require MFA for all users
Users: All users · Cloud apps: All cloud apps
Grant: Require multifactor authentication
Block Legacy Authentication (Critical Addition)
Conditional Access → New Policy:
Name: Block Legacy Authentication
Users: All users
Client Apps: Exchange ActiveSync clients, Other clients
Grant: Block access
Why FIDO2 for admins? AiTM phishing kits intercept both passwords and push MFA tokens in real-time. FIDO2 keys are domain-bound and cannot be phished. Regular users benefit from number matching which hardens against push fatigue, but it is not AiTM-immune. For admin accounts, the risk justifies the hardware investment.
2
Enable Privileged Identity Management (PIM)
CriticalIdentity
4 hrs

No admin should have permanent access to destructive capabilities. PIM forces just-in-time activation with approval. A compromised credential gains nothing without going through the approval workflow.

Configuration Path
Entra ID → Privileged Identity Management → Roles → Intune Administrator
Assignment type: Eligible (NOT Active/Permanent)
Activation maximum duration: 1 hour
Require on activation: MFA + justification + approval
Approver: Security Operations team lead
Notification: security-ops@[company].com on every activation

Apply identical settings to: Global Administrator, Cloud Device Administrator, Security Administrator, Privileged Role Administrator.

PIM requires Entra ID P2 licensing. On P1, minimum alternative: reduce permanent admin accounts to 2 break-glass only, and monitor all admin sign-ins via SIEM. Do NOT make break-glass accounts PIM-eligible — they must always be accessible, including during PIM outages.
3
Enable Continuous Access Evaluation (CAE)
HighIdentity
1 hr

CAE ensures revoked tokens are immediately invalidated rather than remaining valid for 60–90 minutes (the default access token lifetime). Without CAE, if you detect an attack and disable the compromised admin account, their existing session token keeps working for up to 90 minutes — enough time to wipe thousands of devices.

Configuration
Entra ID → Security → Conditional Access → Session → Customize continuous access evaluation
Continuous Access Evaluation: Enabled
Strictly enforce location policies: Enabled (optional for mature environments)
Note on strict location enforcement: This is a more restrictive CAE mode with specific networking requirements (compliant network checks, IP-based location accuracy). Enable standard CAE first and add strict enforcement after validating your named location configuration.
4
Deploy Identity Threat Detection
HighIdentity
4 hrs

Detect AiTM phishing and token theft before they lead to destructive actions. Entra ID Protection provides risk-based signals; SIEM correlation catches behavioral anomalies.

Entra ID Protection
Entra ID → Security → Identity Protection
Sign-in risk policy: Require MFA for medium + high risk
User risk policy: Require password change for high-risk users
KQL — Detect AiTM / Token Replay (adapt to your SIEM schema)
// Detect risky sign-ins indicating possible token replay or AiTM:
SigninLogs
| where ResultType == 0
| where RiskLevelDuringSignIn in ("medium", "high")
| where RiskEventTypes_V2 has_any ("anomalousToken", "tokenIssuerAnomaly", "unfamiliarFeatures")
| project TimeGenerated, UserPrincipalName, AppDisplayName, IPAddress,
          RiskLevelDuringSignIn, RiskEventTypes_V2

// Detect risky admin sign-ins:
SigninLogs
| where RiskState == "atRisk" and UserPrincipalName contains "admin"

// Detect service principal abuse from unknown IPs:
AADServicePrincipalSignInLogs
| where IsInteractive == false and IPAddress !in (known_corp_ips)
Enable Token Protection (preview) to bind tokens to the device they were issued on — pilot in a test group before broad deployment as this feature is not yet generally available.

Restrict Admin Login Locations
CriticalIdentity
2 hrs
Entra ID → Security → Conditional Access → Named Locations
Define: Corporate VPN ranges, SOC network ranges
Block: All locations not in named locations for admin roles
Block: TOR exit nodes, anonymous proxies, high-risk geographies

Create a Conditional Access policy targeting all admin directory roles that blocks access from any location not in the named locations list. This ensures that even with stolen credentials, an attacker logging in from their own infrastructure is blocked before any damage can be done.

2 hrs

Users: All admin roles

7
Harden Break-Glass Accounts
HighIdentity
2 hrs

Break-glass accounts are excluded from Conditional Access by design — this makes them high-value targets. They must also be permanently active (NOT PIM-eligible) so emergency access is guaranteed even during identity service outages.

  • Use passwords of 32+ characters, randomly generated, as a backup credential
  • No mailbox, no phone number. Use phishing-resistant auth (FIDO2 keys or certificate-based). Store FIDO2 keys in a physical safe with dual-custody access, separate from the password
  • Excluded from ALL Conditional Access policies (including the CA Kill Switch — Control 24)
  • Test quarterly. Log every use. Any unplanned login = immediate P1 incident
Alert on ANY Break-Glass Sign-In
SigninLogs
| where UserPrincipalName in ("breakglass1@domain.com", "breakglass2@domain.com")
// Set alert severity: CRITICAL — any sign-in triggers immediate P1 response
Federated & Hybrid Identity Warning: If your M365 tenant federates authentication with Okta, Ping Identity, ADFS, or similar — a compromised upstream IdP bypasses every Entra ID control. Authentication decisions happen before the token reaches Microsoft. See Appendix H for federated identity hardening.
Device Management Controls 8–11
8
Remove Bulk Wipe Capability from Standard Roles
CriticalDevice
2 hrs

This is the single control that most directly prevents a Stryker-style attack. No standard admin role should have bulk wipe permissions. The following Graph API endpoint is what attackers use — make it inaccessible.

POST /deviceManagement/managedDevices/{id}/wipe  ← Intune wipe endpoint
Create Custom Intune Role Without Wipe
Intune Admin Center → Tenant Administration → Roles → Create Custom Role
Role name: Intune Device Manager - No Wipe
Permissions: Enable all EXCEPT:
✗ Remote tasks → Wipe
✗ Remote tasks → Retire
✗ Remote tasks → Factory Reset
✗ Remote tasks → Autopilot Reset
✗ Remote tasks → Delete

Assign this custom role to ALL Intune admins. Reserve wipe capability for break-glass accounts only (permanent active assignment — break-glass must NOT be PIM-eligible).

9
Implement Approval Workflow for Device Wipes
HighDevice
1 day

No destructive action — Wipe, Retire, or Delete — should execute without a second administrator's approval. PIM gates role activation; MAA gates the destructive action itself. An attacker would need to compromise both workflows.

Option A (Recommended): Intune Multi Admin Approval (MAA)
1
Create Approver Security Group: Intune Admin Center → Groups → All Groups → New Group. Add at least two senior security admins. This group must be a member of at least one Intune role assignment.
2
Create Custom Intune Role for Approvers: Intune Admin Center → Tenant Administration → Roles → Create → Intune Role. Grant the "Approval for Multi Admin Approval" permission.
3
Create Access Policy: Intune Admin Center → Tenant Administration → Multi Admin Approval → Access Policies → Create. Set Profile Type to device actions (Wipe, Retire, Delete). Select the approver security group.
Option B (Advanced): Custom Logic App / Power Automate
// Architecture: Azure Logic App or Power Automate
Trigger: HTTP webhook from custom admin portal
→ Send Teams Adaptive Card to Security Ops channel
→ Require approval from 2 members of SG-DeviceWipe-Approvers
→ On approval: Execute Graph API wipe call with full audit logging
→ On rejection: Log and notify requestor
→ Timeout: 30 minutes → auto-reject
Note: MAA does not send built-in notifications. Pair it with a Power Automate flow that monitors the Unified Audit Log for MAA events and sends Teams/email alerts to the approver group.
10
Restrict Device Enrollment and Protect BYOD
HighDevice
1 day

In the Stryker attack, personal phones enrolled in the company portal were also wiped — employees lost personal data, photos, and 2FA authenticator apps. This is entirely preventable. MDM enrollment gives full device control (including full wipe). MAM-only enrollment protects corporate data at the app level without device-level control.

Corporate-Owned Devices
Intune Admin Center → Devices → Enrollment Restrictions
Default platform restriction: Corporate-owned or Windows Autopilot devices
Device limit restriction: 5 devices per user
BYOD / Personal Devices (MAM-Only)
Intune Admin Center → Apps → App Protection Policies → Create Policy
Platform: iOS/iPadOS or Android
Apps: Outlook, Teams, OneDrive (corporate apps)
Data protection: Prevent copy/paste to unmanaged apps
Data protection: Encrypt corporate data within managed apps
Access requirements: Require PIN or biometric
Conditional launch: Block if device is jailbroken/rooted
With MAM-only, a compromised admin can issue a selective wipe (removing only corporate app data) but cannot issue a full device wipe. Personal photos, authenticator apps, and personal data remain untouched. Communicate this to employees as a privacy improvement.
11
Segment Device Management with Intune Scope Tags
HighDevice
1 day

Scope Tags are the blast radius containment control that could have changed the Stryker outcome from catastrophic to contained. Without Scope Tags, a compromised admin can reach every managed device. With Scope Tags, they can only affect devices within their assigned scope — potentially 5,000–10,000 instead of 200,000.

Implementation Steps
1
Create Scope Tags by segment: Intune Admin Center → Tenant Administration → Roles → Scope (Tags)
Create: Region-NorthAmerica, Region-EMEA, Region-APAC, Dept-Engineering, Dept-Finance, etc.
2
Assign Scope Tags to devices: Intune → Devices → All Devices → Select device group → Properties → Scope Tags. Assign appropriate tag based on device location or department.
3
Assign Scope Tags to admin roles: Intune → Tenant Administration → Roles → [Custom Role] → Assignments → Scope (Tags): Select only the tags for the admin's region/department.
Add Entra ID Administrative Units for Identity-Level Segmentation
Entra ID → Roles and Administrators → Administrative Units → Create
Name: AU-NorthAmerica
Members: Add user/device groups for the region
Assign Helpdesk Admin role scoped to this AU only

Disable User Consent

Audit Existing Service Principals and Managed Identities
HighAPI
4 hrs

Controls 12–14 lock down new access. But your tenant likely already has legacy service principals with overprivileged device management permissions granted years ago and forgotten. This is a Day 1 audit.

PowerShell — Enumerate All Service Principals with Device Management Permissions
Connect-MgGraph -Scopes "Application.Read.All"
$apps = Get-MgServicePrincipal -All
$graphSP = Get-MgServicePrincipal -Filter "AppId eq '00000003-0000-0000-c000-000000000000'"

foreach ($app in $apps) {
    $roles = Get-MgServicePrincipalAppRoleAssignment -ServicePrincipalId $app.Id
    $devicePerms = $roles | Where-Object { $_.ResourceDisplayName -eq 'Microsoft Graph' }
    if ($devicePerms) {
        Write-Output "App: $($app.DisplayName) | AppId: $($app.AppId)"
        $devicePerms | ForEach-Object {
            $roleName = ($graphSP.AppRoles | Where-Object { $_.Id -eq $_.AppRoleId }).Value
            Write-Output "  Permission: $roleName"
        }
    }
}
# For each result, ask: Is this app still in use? Does it need DeviceManagement permissions?
# Set a calendar reminder to repeat this audit quarterly
Real risk: Legacy app registrations from pilot projects, former vendors, or decommissioned integrations still hold powerful Graph API permissions. They are invisible to most security reviews because they don't appear in user-facing admin consoles. A single forgotten service principal with DeviceManagementManagedDevices.PrivilegedOperations.All is a backdoor waiting to be exploited.

16
Monitor Mass Device Actions via SIEM with Automated Response
HighDetection
1 day

This is your last line of defense. If all preventive controls fail, this detection rule catches the attack in progress — and the automated playbook stops it within 2 minutes.

KQL — Bulk Wipe Detection (≥5 wipes in 10 min)
// Adapt to your SIEM platform and validate field paths before production deployment
IntuneAuditLogs
| where OperationName in ("wipeDevice", "retireDevice", "deleteDevice")
| extend UPN = tostring(parse_json(Properties).UserPrincipalName)
| summarize WipeCount = count() by UPN, bin(TimeGenerated, 10m)
| where WipeCount >= 5
| project TimeGenerated, UPN, WipeCount
// Set alert severity: CRITICAL — auto-trigger automated response playbook
Automated Response Playbook (5-Step Containment)
// Step 1 — Disable the offending admin account (target: immediate):
Update-MgUser -UserId $attackerUPN -AccountEnabled:$false

// Step 2 — Revoke all active sessions:
Revoke-MgUserSignInSession -UserId $attackerUPN

// Step 3 — Block the source IP in Conditional Access Named Locations:
# PATCH /identity/conditionalAccess/namedLocations/{blocklist-id}

// Step 4 — Send out-of-band alert to IR team:
# Trigger Signal/PagerDuty/OpsGenie with: identity, wipe count, source IP, device scope

// Step 5 — Freeze further destructive actions:
# Temporarily block all wipe/retire/delete via CA by requiring a compliant device claim
# that no current session satisfies
Without automation, typical response time from SIEM alert to account disabled is 30–60 minutes — during which thousands of devices could be wiped. With this playbook, containment happens in under 2 minutes. Test this as part of Simulated Wipe Day (Appendix D).

Configure Audit Log Retention for Forensic Readiness
HighDetection
4 hrs

Standard M365 audit logs are retained for only 180 days (1 year with E5 Compliance). If the attacker had months of dwell time — common in APT operations — you could lose critical forensic evidence before the investigation completes.

With E5 Compliance — Audit (Premium) Retention Policy
Microsoft Purview Compliance Portal → Audit → Audit Retention Policies
Create policy → Duration: 1 year (10-year requires separate add-on license)
Priority: 1 (highest)
Without E5 — Export to Immutable External Storage
# Export Unified Audit Log to external storage on a schedule:
Search-UnifiedAuditLog -StartDate (Get-Date).AddDays(-1) -EndDate (Get-Date) `
    -RecordType AzureActiveDirectory -ResultSize 5000 `
    | Export-Csv -Path "audit-export-$(Get-Date -Format yyyyMMdd).csv"

# Run separate queries for each RecordType (e.g., IntuneOperations)
# For production automation, consider the Management Activity API

# Upload to immutable blob storage (Azure Blob with immutability policy)
# or S3 with Object Lock (AWS) — air-gapped from the M365 tenant
Key principle: Audit logs must survive the attack. If they live exclusively in M365 and M365 is compromised, the attacker may be able to tamper with them. Export to a separate, immutable storage location that requires different credentials.

Network-Level Exfiltration Detection
HighDetection
1 day

Handala claimed to have exfiltrated 50TB of data before executing the wipe (unverified). Data exfiltration will occur through either cloud APIs or network egress points.

Cloud-Layer Detection
  • Enable Unified Audit Log (enabled by default on E3/E5) — captures SharePoint, OneDrive, and Exchange mailbox access
  • Monitor for "FileDownloaded" and "FileSyncDownloadedFull" with abnormally high counts per user per day
  • If you have MCAS (E5 Security): configure anomaly detection for "Mass download" and "Unusual file share activity"
# PowerShell — detect bulk downloads:
Search-UnifiedAuditLog -StartDate (Get-Date).AddDays(-7) -EndDate (Get-Date) `
    -Operations FileDownloaded,FileSyncDownloadedFull `
    -ResultSize 5000 `
    | Group-Object UserIds | Sort-Object Count -Descending | Select -First 20
Internal Network Layer
  • EDR/NDR detects: anomalous data flows, large outbound transfers, C2 beaconing, lateral movement patterns preceding exfiltration
  • EDR on endpoints detects: archive creation (zip/rar of sensitive directories), exfil tools (rclone, MEGAcmd, WinSCP)
  • Without NDR: monitor firewall/proxy logs for single endpoints transferring >1GB outbound, connections to mega.nz, transfer.sh, anonfiles, DNS queries to newly registered domains

Resilience & Recovery Controls 22–24
22
Establish Out-of-Band Emergency Communications
HighResilience
1 day

When all corporate devices are wiped, your Teams, email, Slack, and phone system are gone. Stryker employees could not communicate or access any corporate systems after the wipe — including personal phones enrolled in the company portal. Your IR plan must assume zero corporate infrastructure availability.

  • Maintain an emergency contact list on paper and in a non-corporate messaging app (Signal group for the IR team)
  • Pre-configure an emergency email distribution list on a separate domain/provider not tied to Microsoft 365
  • Store Intune/Entra backup configs and break-glass credentials in a physically secured location
  • Pre-provision emergency communication channels now — do not wait for an incident. Test a dedicated Signal group and emergency Slack workspace on a separate tenant
  • Maintain a printed, offline incident response playbook stored alongside break-glass credentials. Include step-by-step recovery procedures, vendor contact numbers, and escalation paths
  • Conduct a tabletop exercise for: "All corporate devices wiped, zero M365 access"
If you haven't tested this, you don't have a plan. Buy a fireproof safe with dual-combination lock, populate the IR Go-Bag contents (see Appendix B), and assign two keyholders (CISO + IT Director). Test the contents quarterly alongside your break-glass account test.

Conditional Access Kill Switch (Emergency Brake)
HighResilience
1 hr
This is an emergency action, not a routine hardening control. Pre-stage a Conditional Access policy that blocks all admin access to all cloud apps with a single click. Keep it disabled. Enable it only during an active incident when you need to freeze the entire management plane immediately.
Implementation
Entra ID → Security → Conditional Access → New Policy
Name: EMERGENCY_BRAKE_BLOCK_ALL_ADMINS
Users: Include → Directory roles → [All Administrator roles]
Exclude (MANDATORY): Your two break-glass accounts
Target Resources: All Cloud Apps
Conditions: None (apply globally)
Grant: Block Access
Enable Policy: OFF (disabled until needed)
Critical Implementation Rules
  • Break-Glass Exclusion is MANDATORY. If you block all admins without an excluded account, you will be permanently locked out of your tenant, requiring a multi-day identity verification process with Microsoft Support.
  • CAE must be enabled for instant effect. Without CAE (Control 3), an attacker's existing session token could remain valid for up to 90 minutes after you enable this policy.
  • Test in Report-Only mode first. Run it in Report-Only mode for one week. Verify the break-glass exclusion works correctly and confirm you haven't missed service accounts that would break critical automation.
Include this policy in your IR runbook with clear escalation criteria for when to enable it. Define in advance who has authority to pull this lever — typically the CISO or Security Operations lead — and under what conditions. Test quarterly alongside Simulated Wipe Day (Appendix D).

5 Questions to Ask Right Now

Use these to quickly assess your exposure. If your team cannot confidently answer each one, you have work to do.

1
How many accounts in our tenant can wipe devices — and do we know exactly who they are?
Why it matters: In the Stryker attack, a single compromised admin account reportedly wiped 200,000+ devices. Most organizations have far more accounts with wipe permissions than they realize. If you cannot answer this with an exact number today, start with Control 8.
2
Can an Intune administrator access the management portal right now from a personal laptop on public Wi-Fi?
Why it matters: If yes, an attacker who steals admin credentials via phishing can log in from anywhere and execute destructive actions. Conditional Access restricting admin portal access to compliant devices on named networks (Controls 5, 6) closes this vector.
3
Are all privileged roles protected with PIM, requiring approval and justification for every activation?
Why it matters: Permanently assigned admin roles mean an attacker gains destructive capabilities the instant they compromise a credential. PIM adds a human-in-the-loop gate — the attacker would need to request activation through an approval workflow that generates alerts.
4
Is user OAuth application consent disabled across our entire tenant?
Why it matters: Enabled user consent allows attackers to trick employees into granting malicious applications access to device management APIs. A single consent click can give an attacker the Graph API permissions needed to wipe every device in your environment.
5
Do we have SIEM alerts that would fire within minutes if someone attempted to wipe more than 5 devices in a 10-minute window?
Why it matters: The Stryker wipe reportedly affected 200,000+ devices and took hours to complete. An organization with proper monitoring could detect and respond within the first few minutes, limiting damage to a small fraction of what Stryker experienced. The starter KQL query in Control 16 takes 10 minutes to deploy.

Incident Response Physical Survival Kit ("IR Go-Bag")

When every digital system is destroyed, this is what your incident response team reaches for. Store in a fireproof safe with dual-combination lock. Assign two keyholders (CISO + IT Director).

Laminated Break-Glass Credentials
FIDO2 security keys for both break-glass accounts, plus printed 32+ character backup passwords. Store keys and passwords separately under dual-custody access. Laminate to prevent water damage. Update immediately after any use.
Pre-Provisioned FIDO2 Keys
2–4 spare FIDO2 hardware keys, already registered as backup authentication methods for break-glass accounts and key security personnel. Test quarterly.
Encrypted Offline Drive (USB)
Contains: CA-backup.json (Conditional Access policies), Intune-configs.json (device configs), RBAC role definitions, named location definitions, PIM settings. Refresh weekly via scheduled PowerShell export.
Hard Copy Phone Tree
Personal mobile numbers for: CISO, CIO, IT Director, Security Ops lead, Legal counsel, Communications lead, and key vendor contacts. Include Signal group invite QR codes for the emergency channel.
Network Diagram (Printed)
Physical copy of network topology, DNS configuration, VPN endpoints, and ISP contact information. If all systems are wiped, you need this to rebuild network access.
Vendor Emergency Contacts
Microsoft Premier Support case number and PIN, eSentire SOC 24/7 hotline, ISP emergency contact, cyber insurance carrier claims number and policy ID.

The Management Plane vs. Data Plane Detection Gap

Why traditional EDR did not detect the Stryker attack — and where your security tools actually watch.

Identity Plane
Authentication events, token issuance, MFA challenges, risky sign-ins, OAuth consent grants, service principal creation
Entra ID Protection · eSentire MDR for Log
Cloud / Mgmt Plane
API calls to Intune & Graph API: device wipe, retire, delete. Policy changes, role activations, config modifications.
eSentire MDR for Log · eSentire MDR for Cloud
Network Plane
Traffic flows, DNS queries, C2 beaconing, data exfiltration over the wire, lateral movement between endpoints
eSentire MDR for Network · Firewall · Proxy
Endpoint / Data Plane
Malicious files, processes, registry changes, lateral movement on the device itself. Watches what happens ON the device.
eSentire MDR for Endpoint · eSentire MDR Agent
The Gap Stryker Exposed: When a "wipe" command comes from the Intune management plane via a legitimate Graph API call, the EDR on the endpoint treats it as an authorized instruction — not as an attack. The device complies and erases itself. There is no malicious file to detect, no suspicious process to kill, no lateral movement to flag. Detection must happen at the management and identity plane, not the endpoint.

Simulated Wipe Day — Testing Your Defenses

Most security teams have never tested their detection and recovery for a mass-wipe scenario. Run this exercise quarterly. Document results. The first time, things will break — that's the point.

Prerequisites
  • 1 lab/test device enrolled in Intune (NOT a production device)
  • SIEM alert rules from Control 16 deployed
  • CA-backup.json and Intune-configs.json exports from Control 23 available
  • IR team aware this is an exercise
  • Automated playbook from Control 16 in test mode (validate it fires but do not disable production accounts)
Step 1 — Trigger Test Wipe via Graph API
Connect-MgGraph -Scopes "DeviceManagementManagedDevices.PrivilegedOperations.All"
$device = Get-MgDeviceManagementManagedDevice -Filter "deviceName eq 'LAB-TEST-01'"
Invoke-MgGraphRequest -Method POST `
    -Uri "https://graph.microsoft.com/v1.0/deviceManagement/managedDevices/$($device.Id)/wipe" `
    -Body '{}'
Step 2 — Measure Detection and Response Time
  • Start a timer the moment the wipe command is executed
  • Record how long the SIEM alert takes to fire — target: <5 minutes
  • Record how long the automated playbook takes to execute containment — target: <2 minutes after alert
  • If the alert does NOT fire, you have a configuration problem — fix it before a real attack finds it
Step 3 — Test Recovery from Backup
  • Re-enroll the test device via Windows Autopilot or manual enrollment
  • Attempt to restore device configuration from your Intune-configs.json backup
  • Record full cycle time: wipe → re-enroll → compliant device — target: <4 hours
Step 4 — Test Out-of-Band Comms
  • Simulate loss of Teams and corporate email
  • Verify Signal group is active and all IR team members can be reached
  • Verify break-glass credentials from physical safe still work
  • Verify FIDO2 backup keys authenticate successfully

Bulk Device Re-Enrollment Strategy After Mass Wipe

Recovery from a mass wipe is a logistics problem as much as a technical one. Plan this now, not during the crisis.

Phase 1: Triage (Hours 0–24)
  • Activate out-of-band comms (Control 22). Confirm IR team can communicate.
  • Use break-glass accounts to regain tenant access. Disable compromised accounts, revoke all active sessions, rotate credentials.
  • Assess scope: use Intune audit logs or Entra ID sign-in logs to determine blast radius.
  • Restore Conditional Access policies and Intune configurations from backup (Control 23) before re-enrolling any devices.
Phase 2: Prioritized Re-Enrollment (Days 1–7)
  • Priority 1 — Security and IT operations: SOC workstations, IT admin machines, PAWs. Needed to manage the recovery itself.
  • Priority 2 — Executive leadership and legal: Needed for crisis communications and regulatory reporting.
  • Priority 3 — Revenue-critical roles: Sales, customer support, operations staff.
  • Priority 4 — General workforce: Starting with largest regional offices for efficiency.
Phase 3: Enrollment Methods at Scale
# Verify Autopilot registrations survived the wipe (stored in cloud, not on device):
Get-MgDeviceManagementWindowsAutopilotDeviceIdentity | Measure-Object
# If count matches your fleet, mass re-enrollment via Autopilot is dramatically faster.

# For non-Autopilot devices, create Windows Configuration Designer provisioning packages:
# Include: Entra ID join, Intune MDM enrollment URL, Wi-Fi profile
Most important preparation right now: Verify that your Windows Autopilot hardware hashes are registered in the cloud for all corporate devices. Autopilot registrations survive a device wipe because they're stored in the Intune service, not on the device.

Securing Federated and Third-Party Identity Providers

If your M365 tenant federates authentication to Okta, Ping Identity, ADFS, or others — a compromised upstream IdP bypasses every Entra ID control in this guide. Authentication decisions happen before the token reaches Microsoft.

H.1 — Inventory All Federation Trusts
# Discover all federated domains in your tenant:
Get-MgDomain | Where-Object { $_.AuthenticationType -eq "Federated" }
# Pull federation config for each domain:
Get-MgDomainFederationConfiguration -DomainId [domain]
# Any unexpected federation trust is a CRITICAL finding.
H.2 — Monitor Federation Trust Changes in SIEM (Golden SAML Vector)
// Alert on federation configuration changes (how Golden SAML attacks are established):
AuditLogs
| where OperationName in ("Set domain authentication",
    "Set federation settings on domain",
    "Set company information", "Update domain")
H.3 — Require Entra-Native MFA for Admin Roles (Most Important)

Even when federating to Okta or Ping, configure Conditional Access authentication strength policies to require Entra-side FIDO2 for all admin roles. This means even a Golden SAML attack cannot satisfy the MFA requirement.

Conditional Access → Grant → Require authentication strength → Phishing-resistant MFA
(This differs from standard "Require MFA" which may accept the federated MFA claim from your IdP)
H.4 — Audit and Restrict SCIM Provisioning
  • Do NOT sync privileged role assignments via SCIM. Manage all privileged roles exclusively in Entra ID via PIM. The SCIM connector should never be able to assign admin roles.
  • Review the provisioning service principal's permissions quarterly (same process as Control 15). It must not have DeviceManagementManagedDevices.PrivilegedOperations.All.
  • Monitor SCIM write operations. Alert on any provisioning-initiated role changes or group membership modifications to privileged groups.
H.5 — HSM-Protect IdP Signing Keys
  • On-premises ADFS/PingFederate: Store token-signing certificates in an HSM, not in the Windows certificate store. This is Microsoft's primary recommendation to prevent Golden SAML (per Midnight Blizzard / Nobelium attack lessons).
  • Cloud-hosted IdPs (Okta SaaS, Ping Cloud): Verify vendor key management practices, enable key rotation monitoring, and ensure you receive alerts on signing key changes.
H.7 — Restrict Cross-Tenant MFA Trust
Entra ID → External Identities → Cross-tenant access settings → Trust settings
Configure MFA trust only for specific verified partner tenants, not blanket trust for all.
For sensitive resource access, require authentication strength (FIDO2) regardless of partner MFA claims.
Minimum actions for federated environments (under 4 hours): (1) Run H.1 PowerShell to inventory federation trusts; (2) Deploy the H.2 KQL alert in your SIEM; (3) Configure CA authentication strength to require Entra-native FIDO2 for admin roles per H.3.
